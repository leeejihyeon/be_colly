# 백엔드 개발 규칙

## 1. 목적
- 본 문서는 `colly` 백엔드 개발 시 공통 개발/문서화 원칙을 정의한다.
- 모든 개발자는 아래 규칙을 기본 기준으로 적용한다.

## 2. 기본 원칙
- 코드는 가독성과 유지보수성을 최우선으로 작성한다.
- 도메인 간 결합을 최소화하고, 독립적으로 개발/테스트 가능해야 한다.
- 단기 구현보다 확장 가능한 구조를 우선한다.
- 구현 변경 시 문서(API/DDL/TODO)도 함께 갱신한다.

## 3. 설계 원칙
- 구조는 도메인 중심 + Port/Adapter(헥사고날) 패턴을 따른다.
- 의존 방향은 아래를 고정한다.
  - `adapter.in -> application`
  - `application -> domain + application.port.out`
  - `adapter.out -> application.port.out + shared`
  - `domain`은 다른 계층을 알지 않는다.
- JPA Entity는 도메인 경계 밖으로 직접 노출하지 않는다.

## 4. 문서 폴더 구조 규칙
- 문서 루트는 `doc`를 사용한다.
- 문서는 아래 3개 루트 폴더로 관리한다.
  - `doc/planning`: 기획 문서
  - `doc/api`: 도메인별 API 명세 문서
  - `doc/ddl`: 실제 사용하는 스키마 DDL 원본 소스
- 본 문서는 개발자/리뷰어가 함께 참고하는 기준 문서이다.

## 4.1 각 폴더의 역할
- `doc/planning`
  - 서비스 요구사항, 정책, 화면/플로우 기반 기획 내용을 기록한다.
  - 상위 기획서는 `전체-기획서.md`로 관리한다.
- `doc/api`
  - 도메인별로 폴더를 만들고 API 명세서를 관리한다.
  - 예: `doc/api/user/사용자-API-명세서.md`, `doc/api/order/주문-API-명세서.md`
  - 공통 규칙 문서는 `doc/api/common/공통-API-명세서.md`를 기준으로 한다.
- `doc/ddl`
  - 운영/개발에서 기준이 되는 테이블 DDL 원본 SQL을 보관한다.
  - 테이블 단위 파일을 권장한다. 예: `user.sql`, `order.sql`

## 5. API 문서화 규칙
- API 구현이 완료되면 `doc/api` 하위 도메인 문서를 반드시 갱신한다.
- 각 API는 아래 항목을 포함한다.
  - 엔드포인트(Method, Path)
  - 요청 스펙(Header, Path Variable, Query, Body)
  - 응답 스펙(성공/실패 예시)
  - 에러 코드 및 조건
  - 호출 맥락(어떤 기능/플로우에서 호출되는지)
- 구현 코드와 문서가 다르면 구현 변경 시점에 문서를 함께 수정한다.

## 6. DDL 관리 규칙
- `doc/ddl`의 SQL은 실제 운영 기준 스키마와 동기화 상태를 유지한다.
- 테이블 구조 변경 시 아래 순서로 반영한다.
  1. 마이그레이션(Flyway 등) 반영
  2. `doc/ddl` 원본 SQL 갱신
  3. 관련 API 문서/기획 문서 영향도 갱신

## 7. TODO 운영 규칙
- 미완료 항목은 `doc/TODO-작업-목록.md`에서 중앙 관리한다.
- TODO를 추가/완료할 때 본 파일을 함께 갱신한다.
- 배포 전 TODO 미완료 항목을 점검한다.

## 8. 주석/문서화 규칙 (필수)
- Java 코드는 클래스, 메서드, DTO(요청/응답 타입)에 문서형 주석(`JavaDoc`)을 작성한다.
- `application.port.in`의 모든 `*UseCase` 인터페이스는 인터페이스/메서드/명령·결과 record까지 JavaDoc을 작성한다.
- JPA 엔티티의 공개 생성자에는 목적 설명과 모든 파라미터 `@param` JavaDoc을 작성한다.
- `adapter.in.web` 계층의 요청/응답 DTO는 컨트롤러 내부 클래스(record)로 두지 않고 `adapter.in.web.dto` 패키지의 독립 파일로 분리한다.
- 컨트롤러는 라우팅/유스케이스 호출에 집중하고, DTO 정의/변환 책임은 `dto` 클래스로 분리한다.
- 메서드 주석 규칙은 접근제어자와 무관하게 적용한다.
  - `public`, `protected`, `package-private`, `private` 모두 동일 적용
- 메서드 주석에는 아래 항목을 포함한다.
  - 목적/동작 설명
  - `@param` (모든 파라미터)
  - `@return` (반환값이 있는 경우)
  - `@throws` (도메인 예외를 던지는 경우 권장)
- DTO/record 타입은 필드 의미를 `@param`으로 문서화한다.

## 9. JPA 엔티티 규칙
- 엔티티 클래스명은 `JpaEntity` 접미사를 사용하지 않고, 테이블 의미를 반영한 이름을 사용한다.
  - 예: `users` 테이블 -> `UserEntity`, `community_post` 테이블 -> `CommunityPostEntity`
- 엔티티 getter는 수동 구현 대신 Lombok `@Getter`를 기본 사용한다.
- JPA 기본 생성자는 수동 구현 대신 Lombok `@NoArgsConstructor(access = AccessLevel.PROTECTED)`를 기본 사용한다.
- 엔티티 생성 책임이 드러나도록 공개 생성자는 명시적으로 유지하고 JavaDoc으로 목적/파라미터를 설명한다.
