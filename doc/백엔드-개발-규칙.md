# 백엔드 개발 규칙

## 1. 목적
- 본 문서는 `colly` 백엔드 개발 시 공통 개발/문서화 원칙을 정의한다.
- 모든 개발자는 아래 규칙을 기본 기준으로 적용한다.

## 2. 기본 원칙
- 코드는 가독성과 유지보수성을 최우선으로 작성한다.
- 도메인 간 결합을 최소화하고, 독립적으로 개발/테스트 가능해야 한다.
- 단기 구현보다 확장 가능한 구조를 우선한다.
- 구현 변경 시 문서(API/DDL/TODO)도 함께 갱신한다.

## 3. 설계 원칙
- 구조는 도메인 중심 + Port/Adapter(헥사고날) 패턴을 따른다.
- 의존 방향은 아래를 고정한다.
  - `adapter.in -> application`
  - `application -> domain + application.port.out`
  - `adapter.out -> application.port.out + shared`
  - `domain`은 다른 계층을 알지 않는다.
- JPA Entity는 도메인 경계 밖으로 직접 노출하지 않는다.

## 4. 문서 폴더 구조 규칙
- 문서 루트는 `doc`를 사용한다.
- 문서는 아래 3개 루트 폴더로 관리한다.
  - `doc/planning`: 기획 문서
  - `doc/api`: 도메인별 API 명세 문서
  - `doc/ddl`: 실제 사용하는 스키마 DDL 원본 소스
- 본 문서는 개발자/리뷰어가 함께 참고하는 기준 문서이다.

## 4.1 각 폴더의 역할
- `doc/planning`
  - 서비스 요구사항, 정책, 화면/플로우 기반 기획 내용을 기록한다.
  - 상위 기획서는 `전체-기획서.md`로 관리한다.
- `doc/api`
  - 도메인별로 폴더를 만들고 API 명세서를 관리한다.
  - 예: `doc/api/user/사용자-API-명세서.md`, `doc/api/order/주문-API-명세서.md`
  - 공통 규칙 문서는 `doc/api/common/공통-API-명세서.md`를 기준으로 한다.
- `doc/ddl`
  - 운영/개발에서 기준이 되는 테이블 DDL 원본 SQL을 보관한다.
  - 테이블 단위 파일을 권장한다. 예: `user.sql`, `order.sql`

## 5. API 문서화 규칙
- API 구현이 완료되면 `doc/api` 하위 도메인 문서를 반드시 갱신한다.
- 각 API는 아래 항목을 포함한다.
  - 엔드포인트(Method, Path)
  - 요청 스펙(Header, Path Variable, Query, Body)
  - 응답 스펙(성공/실패 예시)
  - 에러 코드 및 조건
  - 호출 맥락(어떤 기능/플로우에서 호출되는지)
- 구현 코드와 문서가 다르면 구현 변경 시점에 문서를 함께 수정한다.

## 6. DDL 관리 규칙
- `doc/ddl`의 SQL은 실제 운영 기준 스키마와 동기화 상태를 유지한다.
- 테이블 구조 변경 시 아래 순서로 반영한다.
  1. 마이그레이션(Flyway 등) 반영
  2. `doc/ddl` 원본 SQL 갱신
  3. 관련 API 문서/기획 문서 영향도 갱신
- 스키마 정의 시 FK/인덱스 규칙을 기본 적용한다.
  - 참조 무결성이 필요한 관계 컬럼은 `FOREIGN KEY`를 명시한다.
  - 주요 조회 조건(WHERE/JOIN/ORDER BY)에 사용되는 컬럼은 목적에 맞는 인덱스를 추가한다.
  - 유니크 보장이 필요한 식별 조합은 `UNIQUE KEY` 또는 `UNIQUE CONSTRAINT`를 사용한다.

## 7. TODO 운영 규칙
- 미완료 항목은 `doc/TODO-작업-목록.md`에서 중앙 관리한다.
- TODO를 추가/완료할 때 본 파일을 함께 갱신한다.
- 배포 전 TODO 미완료 항목을 점검한다.

## 8. 주석/문서화 규칙 (필수)
- Java 코드는 클래스, 메서드, DTO(요청/응답 타입)에 문서형 주석(`JavaDoc`)을 작성한다.
- `application` 계층(`application.service`, `application.port.in`, `application.port.out`)은 클래스/인터페이스/메서드 JavaDoc을 작성한다.
- `application.port.in`의 `record` 응답/명령 타입(`*View`, `*Command`, `*Result`, `*Query`)도 JavaDoc을 작성한다.
- `application.port.in`의 모든 `*UseCase` 인터페이스는 인터페이스/메서드/명령·결과 record까지 JavaDoc을 작성한다.
- `application.port.out` 인터페이스는 인터페이스/메서드에 JavaDoc을 작성한다.
- `domain` 계층(`domain.model`, `domain.policy`)은 클래스/enum/메서드 JavaDoc을 작성한다.
  - 정적 팩토리 메서드(`create`, `restore`)와 도메인 행위 메서드도 반드시 작성한다.
  - 적용 범위는 `community`, `order`, `auth`, `user` 등 모든 도메인에 동일하게 적용한다.
- JPA 엔티티의 공개 생성자에는 목적 설명과 모든 파라미터 `@param` JavaDoc을 작성한다.
- `adapter.out.persistence.repository`의 리포지토리 인터페이스는 인터페이스/쿼리 메서드에 JavaDoc을 작성한다.
- `adapter` 계층(`adapter.in`, `adapter.out`, `adapter.out.mapper`) 클래스는 클래스 설명 JavaDoc을 작성한다.
- `adapter` 계층의 메서드는 public/private 구분 없이 JavaDoc을 작성한다.
  - 목적/동작, `@param`, `@return`을 기본 포함한다.
- `adapter.in.web` 계층의 요청/응답 DTO는 컨트롤러 내부 클래스(record)로 두지 않고 `adapter.in.web.dto` 패키지의 독립 파일로 분리한다.
- 컨트롤러는 라우팅/유스케이스 호출에 집중하고, DTO 정의/변환 책임은 `dto` 클래스로 분리한다.
- `Request`, `Response` 접미사는 `adapter.in.web.dto`의 HTTP 요청/응답 DTO 타입에서만 사용한다.
  - UseCase/Service/Domain/Port 타입명에는 `Request`, `Response` 접미사를 사용하지 않는다.
- 메서드 주석 규칙은 접근제어자와 무관하게 적용한다.
  - `public`, `protected`, `package-private`, `private` 모두 동일 적용
- 메서드 주석에는 아래 항목을 포함한다.
  - 목적/동작 설명
  - `@param` (모든 파라미터)
  - `@return` (반환값이 있는 경우)
  - `@throws` (도메인 예외를 던지는 경우 권장)
- DTO/record 타입은 필드 의미를 `@param`으로 문서화한다.

## 9. JPA 엔티티 규칙
- 엔티티 클래스명은 `JpaEntity` 접미사를 사용하지 않고, 테이블 의미를 반영한 이름을 사용한다.
  - 예: `users` 테이블 -> `UserEntity`, `community_post` 테이블 -> `CommunityPostEntity`
- 서로 다른 패키지에서 동일 클래스명 엔티티를 사용할 경우, `@Entity(name = "...")`로 엔티티 이름을 고유하게 지정한다.
- 엔티티 getter는 수동 구현 대신 Lombok `@Getter`를 기본 사용한다.
- JPA 기본 생성자는 수동 구현 대신 Lombok `@NoArgsConstructor(access = AccessLevel.PROTECTED)`를 기본 사용한다.
- 엔티티 생성 책임이 드러나도록 공개 생성자는 명시적으로 유지하고 JavaDoc으로 목적/파라미터를 설명한다.

## 10. JPA 운영 규칙
- 연관관계 매핑 기본값은 `LAZY`로 둔다.
  - 양방향 매핑은 꼭 필요한 경우에만 사용하고, 기본은 단방향 또는 ID 참조를 우선한다.
- N+1 방지 규칙을 적용한다.
  - 목록/상세 조회에서 연관 데이터가 필요한 경우 `fetch join` 또는 `@EntityGraph`를 우선 검토한다.
  - 조회 성능 이슈가 있는 API는 쿼리 수를 점검하고 개선 근거를 남긴다.
- 복잡 조회 구현은 Querydsl을 기본 사용한다.
  - 단순 CRUD 파생 쿼리는 Spring Data 메서드명을 사용해도 된다.
  - Querydsl로도 비효율적이거나 표현이 어려운 경우에만 네이티브 SQL을 사용한다.
  - 네이티브 SQL 사용 시 선택 사유와 성능 근거를 코드/문서에 남긴다.
- 감사 컬럼(`created_at`, `updated_at`)은 `BaseEntity` + JPA Auditing으로 자동 관리한다.
  - 엔티티/서비스 코드에서 감사 컬럼을 수동으로 세팅하지 않는다.
- 스키마 관리는 Flyway를 기준으로 한다.
  - `ddl-auto`는 `none`으로 유지한다.
  - 스키마 변경 시 순서는 `Flyway 마이그레이션` -> `doc/ddl 갱신` -> `API/기획/TODO 영향도 반영`을 지킨다.

## 11. Enum 규칙
- 모든 enum은 한글/영문 의미 라벨을 함께 가진다.
  - 필수 필드: `koLabel`, `engLabel`
  - 필수 접근자: `getKoLabel()`, `getEngLabel()`
- enum 상수 추가/변경 시 라벨도 함께 갱신한다.
- DB 스키마 타입은 `ENUM`을 사용하지 않고 `VARCHAR`를 사용한다.
  - enum 값 제약은 애플리케이션 레벨에서 관리한다.
  - JPA 매핑이 필요한 경우 `@Enumerated(EnumType.STRING)` 또는 값 변환 로직으로 문자열 저장을 사용한다.

## 12. Lombok 규칙
- 보일러플레이트 코드(생성자 주입, getter)는 Lombok을 기본 사용한다.
  - 주입 전용 클래스(Controller/Service/Adapter)는 `@RequiredArgsConstructor`를 기본 사용한다.
  - 값 조회 위주 모델/엔티티는 `@Getter`를 기본 사용한다.
- 수동 getter/생성자는 특별한 이유가 있을 때만 사용하고, 사용 시 코드 주석으로 이유를 남긴다.
