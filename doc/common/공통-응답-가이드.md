# 공통 응답 가이드

## 1. 왜 필요한가
- 프론트엔드가 API마다 다른 응답 형태를 처리하지 않도록 응답 포맷을 통일하기 위함.
- 운영/디버깅 시 `code`, `timestamp` 등 공통 메타 정보를 일관되게 제공하기 위함.

## 2. 현재 기준 구성 요소
| 클래스 | 경로 | 역할 |
|---|---|---|
| `ApiResponse<T>` | `src/main/java/lab/coder/colly/shared/api/ApiResponse.java` | 성공 응답 공통 바디 |
| `ApiErrorResponse` | `src/main/java/lab/coder/colly/shared/api/ApiErrorResponse.java` | 실패 응답 공통 바디 |
| `ApiResponses` | `src/main/java/lab/coder/colly/shared/api/ApiResponses.java` | 컨트롤러용 응답 생성 헬퍼 |

## 3. 언제/어디서 사용하나
- 언제:
  - 모든 REST API 성공 응답 생성 시
  - 전역 예외 처리에서 실패 응답 생성 시
- 어디서:
  - `adapter.in.web` 컨트롤러: `ApiResponses.ok/created(...)`
  - `GlobalExceptionHandler`: `ApiErrorResponse.failure(...)`

## 4. 어떻게 사용하나
- 컨트롤러 반환 타입 표준:
  - `ResponseEntity<ApiResponse<T>>`
- 성공 응답 생성:
  - 조회: `ApiResponses.ok(data)`
  - 생성: `ApiResponses.created(data)`
- 실패 응답 생성:
  - 예외 핸들러에서 `ApiErrorResponse.failure(code, message, path, details)`

## 5. 응답 스키마
### 5.1 성공
```json
{
  "success": true,
  "code": "OK",
  "message": "요청이 성공했습니다.",
  "data": {},
  "timestamp": "2026-03-27T14:30:00"
}
```

### 5.2 실패
```json
{
  "success": false,
  "code": "ERROR_CODE",
  "message": "상세 메시지",
  "path": "/api/example",
  "details": {},
  "timestamp": "2026-03-27T14:30:00"
}
```

## 6. 팀 규칙
- 컨트롤러에서 `Map`을 직접 반환하지 않는다.
- 컨트롤러에서 도메인별 임의 응답 래퍼를 새로 만들지 않는다.
- 상태코드는 `ResponseEntity`로 제어하고, 응답 바디는 `ApiResponse/ApiErrorResponse`로 통일한다.
