# 공통 에러 처리 가이드

## 1. 왜 필요한가
- 예외 처리 책임을 컨트롤러 밖으로 분리해 도메인/컨트롤러 코드를 단순화하기 위함.
- 에러 코드와 HTTP 상태 매핑을 중앙화해 API 일관성을 유지하기 위함.

## 2. 현재 기준 구성 요소
| 클래스 | 경로 | 역할 |
|---|---|---|
| `ErrorCode` | `src/main/java/lab/coder/colly/shared/error/ErrorCode.java` | 도메인 에러 코드 + 한글/영문 라벨 |
| `DomainException` | `src/main/java/lab/coder/colly/shared/error/DomainException.java` | 비즈니스 예외 표현 |
| `GlobalExceptionHandler` | `src/main/java/lab/coder/colly/shared/error/GlobalExceptionHandler.java` | 예외를 HTTP 응답으로 변환 |

## 3. 언제/어디서 사용하나
- 언제:
  - 비즈니스 정책 위반/리소스 미존재/권한 오류 발생 시
  - 요청 값 검증 실패 시
  - 처리되지 않은 런타임 예외 발생 시
- 어디서:
  - `domain`, `application.service` 계층: `throw new DomainException(...)`
  - `shared.error.GlobalExceptionHandler`: 공통 실패 응답 변환

## 4. 어떻게 사용하나
- 비즈니스 예외 발생:
  - `throw new DomainException(ErrorCode.INVALID_AMOUNT, "Order amount must be greater than 0")`
- HTTP 매핑:
  - `GlobalExceptionHandler`에서 `ErrorCode`별 `HttpStatus`를 고정 매핑
- Validation 예외:
  - `MethodArgumentNotValidException`을 `VALIDATION_ERROR`로 변환
  - `details.fieldErrors`에 필드별 오류 목록 포함
- 예상치 못한 예외:
  - `INTERNAL_SERVER_ERROR`(500)로 변환

## 5. 팀 규칙
- `ErrorCode`는 에러 의미가 겹치지 않게 관리한다.
- 신규 `ErrorCode` 추가 시:
  1. `ErrorCode` enum 추가
  2. `GlobalExceptionHandler` 상태코드 매핑 추가
  3. `doc/api/common/공통-API-명세서.md` 에러 코드 표 업데이트
- 서비스/도메인에서 HTTP 상태코드를 직접 다루지 않는다.
